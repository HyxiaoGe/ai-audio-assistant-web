# 会议摘要优化分析报告

## 一、当前实现分析

### 1.1 现有架构概览

当前系统生成**三种类型**的摘要：
- **Overview (内容概述)** - 200字简要总结
- **Key Points (关键要点)** - 5-10个核心观点
- **Action Items (待办事项)** - 可执行的任务列表

支持**5种内容风格**：
- `meeting` - 会议、讨论、头脑风暴
- `lecture` - 讲座、课程、培训
- `podcast` - 播客、访谈、对话节目
- `video` - 视频教程、解说视频
- `general` - 通用内容

### 1.2 当前提示词模板分析

**优点：**
✅ 清晰的角色定位（系统角色基于内容风格）
✅ 明确的格式要求（Markdown列表、任务列表）
✅ 温度参数调优（Overview 0.7, Key Points 0.5, Action Items 0.3）
✅ 区分信息点和行动项的指导明确

**不足：**
❌ **缺少结构化章节** - 没有按主题或时间线组织内容
❌ **缺少发言人统计** - 未充分利用speaker diarization数据
❌ **缺少决策和共识记录** - 没有明确标注决议内容
❌ **缺少问题与风险识别** - 未提取未解决问题和潜在风险
❌ **缺少时间戳关联** - 无法快速定位到原始会议片段
❌ **缺少优先级标注** - 待办事项没有优先级分类
❌ **输出格式单一** - 纯文本，缺少可视化元素（表格、流程等）

## 四、优化建议方案

### 4.1 短期优化（保持现有架构）

**方案：增强现有提示词，无需改动数据模型**

#### 优化目标
在不改动数据库schema和API的前提下，通过优化提示词提升输出质量。

#### 具体改进

**A. Overview (内容概述) 优化**

**当前问题：**
- 缺少会议元数据
- 缺少结构化章节

**优化后提示词：**
```
请为以下会议转写文本生成一份结构化的内容概述（500字以内）：

{transcript}

输出要求：

【会议概览】
用2-3句话总结会议核心内容和主要成果

【讨论主题】
按时间顺序或重要性列出3-5个主要讨论主题：
1. 主题A：简要说明讨论内容和结论
2. 主题B：简要说明讨论内容和结论
3. ...

【核心决策】
列出会议中明确做出的决策或达成的共识（如果有）：
- 决策1
- 决策2

【关键数据】（如果转写中提到数字、指标等）
- 数据1
- 数据2

格式要求：
1. 使用清晰的分段标题（【】包裹）
2. 保持简洁专业
3. 突出行动导向的结论
4. 如果某个部分没有相关内容，可以省略该部分
```

**参数调整：**
```json
{
  "temperature": 0.6,  // 从0.7降到0.6，更结构化
  "max_tokens": 1000,  // 从500提升到1000，支持更丰富内容
  "top_p": 0.9
}
```

---

**B. Key Points (关键要点) 优化**

**当前问题：**
- 缺少分类组织
- 缺少发言人上下文
- 缺少优先级排序

**优化后提示词：**
```
请从以下会议转写文本中提取关键要点（8-15个），并按类别组织：

{transcript}

输出要求：

【决策与共识】（如果有）
- 决策1：具体决定的内容 [发言人: 张三]
- 共识2：团队达成一致的观点 [多人讨论]

【重要信息与数据】
- 关键信息1：事实性描述 [发言人: 李四]
- 数据指标2：具体数字和分析

【核心观点】
- 观点1：某人提出的重要看法 [发言人: 王五]
- 观点2：有价值的建议或想法

【问题与挑战】（如果讨论中提到）
- 问题1：当前面临的困难
- 风险2：需要关注的潜在风险

格式要求：
1. 每个要点一句话精炼表达
2. 在要点后用[发言人: XX]标注（如果转写中有speaker信息）
3. 按重要性在每个类别内排序
4. 使用「-」开头的Markdown列表
5. 如果某个类别没有相关内容，省略该类别
6. 区分「信息」和「行动」：信息放这里，行动放待办事项
```

**参数调整：**
```json
{
  "temperature": 0.5,  // 保持不变
  "max_tokens": 1200,  // 从800提升到1200
  "top_p": 0.9,
  "frequency_penalty": 0.3  // 从0.2提升到0.3，减少重复
}
```

---

**C. Action Items (待办事项) 优化**

**当前问题：**
- 缺少优先级
- 缺少类别分组
- 缺少未解决问题

**优化后提示词：**
```
请从以下会议转写文本中提取所有待办事项、行动项和未解决问题：

{transcript}

输出要求：

【待办事项】

高优先级（紧急重要）：
- [ ] 任务描述 @负责人 (截止时间) - 简要说明为什么重要

普通优先级：
- [ ] 任务描述 @负责人 (截止时间)
- [ ] 任务描述 @负责人 (如果没有明确截止时间)

低优先级（可选）：
- [ ] 任务描述

【未解决问题】
如果会议中有明确提出但未得到解决的问题：
1. 问题描述 - 需跟进人: @XX
2. 问题描述 - 需跟进人: @YY

【下一步行动】
如果会议中提到后续会议或跟进计划：
- 后续行动的描述

提取规则：
1. 只提取明确的、可执行的行动项（有具体动作动词）
2. 优先级判断标准：
   - 高优先级：会议中明确强调紧急/重要，或有近期截止时间
   - 普通优先级：常规工作任务
   - 低优先级：会议中提到"有空就做""可以考虑"等
3. 格式：- [ ] 任务描述 @负责人 (截止时间)
4. 区分「信息」和「行动」：\"市场份额下降了\"是信息，\"分析市场份额下降原因\"才是行动项
5. 如果转写中包含speaker信息，优先提取作为负责人
6. 如果某个部分没有相关内容，输出「暂无XXX」
```

**参数调整：**
```json
{
  "temperature": 0.3,  // 保持不变，保证准确性
  "max_tokens": 1000,  // 从600提升到1000
  "top_p": 0.9
}
```

## 七、关键决策点

在实施前，需要明确以下问题：

### 决策1：是否接受LLM成本增加？

- **短期优化：+30%**
- **中期优化：+100%**
- **长期优化：+150%**

如果成本敏感，可以：
- 仅对付费用户启用增强版摘要
- 提供"标准版"和"专业版"两种摘要模式
- 使用更便宜的模型（如qwen）生成部分内容

### 决策2：数据模型改造的激进程度？

- **保守：** 利用现有content字段存JSON
- **激进：** 创建专门的表结构

推荐：**保守方案**，因为：
- 快速迭代
- 向后兼容
- 降低风险

### 决策3：优先级排序？

基于业务目标选择：

**如果目标是快速改善用户体验：**
→ 短期优化 → 中期优化（路径A）

**如果目标是构建差异化竞争力：**
→ 短期优化 → 中期优化 → 长期优化

**如果目标是快速验证市场反馈：**
→ 仅短期优化，收集用户反馈后决定下一步

## 总结

当前系统的会议摘要功能具有良好的基础架构，但在输出格式和内容丰富度上与业界主流产品（钉钉、豆包、飞书、腾讯会议）存在差距。

**核心差距：**
1. 缺少结构化章节组织
2. 缺少会议元数据和发言人统计
3. 缺少决策记录和问题追踪
4. 缺少可视化元素
5. 缺少时间戳关联

**推荐路线：**
- **立即实施：** 短期优化（优化提示词）
- **1-2周后：** 中期优化（结构化存储）
- **2-3月后：** 长期优化（可视化与交互）

通过分阶段实施，可以在控制风险和成本的前提下，逐步提升产品竞争力，最终达到业界一流水平。
